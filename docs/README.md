## Documentation Philosophy

### Single Source of Truth

All documentation for a specific feature is consolidated into a single comprehensive guide. This approach:

- **Reduces fragmentation**: No need to search multiple files
- **Improves discoverability**: Everything in one place
- **Easier maintenance**: Update one file instead of many
- **Better for searching**: Search within a single document
- **Clearer navigation**: Table of contents for quick access

### When to Create New Documentation

Create a new documentation file when:

1. **New major feature**: A significant new feature with multiple aspects
2. **Different audience**: Documentation for different user types (developers vs users)
3. **Different purpose**: Setup guides vs API reference vs tutorials

### When to Update Existing Documentation

Update existing documentation when:

1. **Feature enhancement**: Adding to existing functionality
2. **Bug fixes**: Correcting errors or clarifications
3. **New examples**: Adding usage examples or workflows
4. **Design changes**: Updating architecture or design decisions

## Documentation Structure

Each comprehensive guide should include:

1. **Table of Contents**: For easy navigation
2. **Quick Start**: Get up and running in 5 minutes
3. **Overview**: High-level architecture and concepts
4. **Setup Guide**: Detailed setup instructions
5. **Usage Examples**: Common workflows and patterns
6. **API Reference**: Technical details and interfaces
7. **Design Decisions**: Why things work the way they do
8. **Troubleshooting**: Common issues and solutions

## Contributing to Documentation

When updating documentation:

1. **Keep it current**: Update docs when code changes
2. **Be comprehensive**: Cover all aspects in one place
3. **Use examples**: Show, don't just tell
4. **Explain why**: Include design decisions and rationale
5. **Add troubleshooting**: Document common issues you encounter
